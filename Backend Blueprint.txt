LOXX BACKEND BLUEPRINT
================================================================================
v1.0 (Enterprise Edition)
Professional Backend Architecture & Implementation Plan

This document provides a comprehensive architectural blueprint for the LOXX 
platform backend, integrating frontend requirements, technical specifications, 
and integration protocols.

================================================================================
1. SYSTEM ARCHITECTURE
================================================================================

The architecture is designed as a distributed, high-performance system capable of 
handling tens of thousands of concurrent users with ultra-low latency.

- API Server: Node.js (TypeScript) + Express.js handling stateless REST requests.
- Realtime WebSocket Gateway: Socket.io instances grouped in a cluster using 
  Redis Adapter for cross-server event synchronization.
- WebRTC Signaling Service: Specialized signaling layer for Mediasoup SFU 
  coordination, managing media transports and producers.
- Database Layer: 
    * PostgreSQL (Primary): Master-Slave replication. Master for writes, 
      Read Replicas for high-traffic queries (Leaderboards, Games list).
- Caching & State Layer: 
    * Redis Cluster: Used for session storage, presence tracking, anti-spam 
      buckets, and lobby state (HashMaps).
- Notification Service: Event-driven worker that pushes persistent notifications 
  to DB and real-time alerts to WebSockets.
- Media Storage: S3-compatible object storage (e.g., MinIO or AWS S3) for 
  avatars and banners, protected via Signed URLs.

================================================================================
2. CORE SERVICES
================================================================================

- Auth Service: Manages identity, Argon2id hashing, and JWT lifecycle 
  (Rotation/Blacklisting).
- User Service: Handles profile metadata, Persian username validation, and S3 
  upload coordination.
- Friendship Service: Manages the social graph, blocking logic, and presence 
  broadcasting.
- Chat Service: Manages text persistence, message threading (replies), 
  mentions, and reactions.
- Lobby Service: Orchestrates group formation, matchmaking logic, and 
    lobby lifecycle (auto-close).
- Voice Signaling Service: High-performance bridge between WebSockets and 
  Mediasoup Workers.
- Notification Service: Centralized hub for delivering social and system alerts.
- Ranking Service: XP calculation engine and scheduled leaderboard aggregation.
- Subscription Service: Commercial gatekeeper for PLUS and VIP tier features.

================================================================================
3. DATA MODELS (POSTGRESQL)
================================================================================

User (Table: users)
- id: UUID (PK)
- username: VARCHAR(255) UNIQUE (Supports Persian/UTF-8)
- email: VARCHAR(255) UNIQUE
- password_hash: TEXT (Argon2id)
- is_verified: BOOLEAN
- created_at: TIMESTAMP
- INDEX: username, email

Profile (Table: profiles)
- user_id: UUID (FK -> Users)
- display_name: VARCHAR(255)
- bio: TEXT (Persian)
- avatar_url: TEXT
- banner_url: TEXT
- level: INT (DEFAULT 1)
- xp: INT (DEFAULT 0)
- membership_type: ENUM ('none', 'plus', 'vip')
- region: ENUM ('IR', 'TR', 'EU', 'ME')
- last_activity: TIMESTAMP

Friendship (Table: friendships)
- requester_id: UUID (FK)
- target_id: UUID (FK)
- status: ENUM ('pending', 'accepted', 'blocked')
- is_favorite: BOOLEAN
- is_muted: BOOLEAN
- created_at: TIMESTAMP
- INDEX: (requester_id, target_id)

Message (Table: messages)
- id: BIGINT (PK)
- channel_id: VARCHAR(64)
- sender_id: UUID (FK)
- content: TEXT (Persian/UTF-8)
- reply_to_id: BIGINT (FK)
- reactions: JSONB
- metadata: JSONB (GIF URLs, Lobby Invites)
- created_at: TIMESTAMP
- INDEX: channel_id, sender_id

Lobby & Members (Tables: lobbies, lobby_members)
- lobby: id, game_id, title, host_id, max_players, password, region, status.
- members: lobby_id, user_id, is_ready, mic_status.

Notification (Table: notifications)
- id: UUID
- user_id: UUID
- type: VARCHAR
- data: JSONB
- is_read: BOOLEAN
- created_at: TIMESTAMP

XPLog (Table: xp_logs)
- id: BIGINT
- user_id: UUID
- source: ENUM ('lobby_join', 'match_start', 'chat_daily', 'invite_friend')
- amount: INT
- timestamp: TIMESTAMP

4. REST API DESIGN
================================================================================

Domain: Auth
- POST /api/auth/register (username, email, password)
- POST /api/auth/login (email, password)
- POST /api/auth/refresh (silent rotation)
- POST /api/auth/logout (blacklist token)

Domain: Profile
- GET /api/profile/me
- GET /api/profile/:username
- PATCH /api/profile/update (bio, display_name, region)
- POST /api/profile/avatar (multipart/form-data)

Domain: Friends
- GET /api/friends/list (Cursor pagination)
- POST /api/friends/request (target_id)
- POST /api/friends/respond (request_id, action)

Domain: Chat
- GET /api/chat/messages/:channelId (?cursor)
- POST /api/chat/send (content, channel_id, reply_to)

Domain: Lobbies
- GET /api/lobbies (?game_id, ?region)
- POST /api/lobbies/create
- POST /api/lobbies/:id/join

Domain: Ranking
- GET /api/ranking (?period: weekly|monthly|all-time)

================================================================================
5. REALTIME WEBSOCKET EVENTS
================================================================================

Namespace: /platform

User Presence:
- Event: 'user:presence_update'
- Payload: { userId, status: 'online'|'away'|'in-game', activity: 'CS2' }
- Target: All friends of the user.

Communication:
- Event: 'chat:message'
- Payload: { id, sender, text, channelId, replyTo, reactions }
- Target: All users in the channel.
- Event: 'chat:typing'
- Payload: { userId, channelId }
- Target: All users in the channel (excluding sender).

Lobby Synchronization:
- Event: 'lobby:update'
- Payload: { lobbyId, status, playersCount, memberList }
- Target: All users in the lobby room.
- Event: 'lobby:countdown'
- Payload: { lobbyId, secondsRemaining }
- Target: Members of the specific lobby.

================================================================================
6. VOICE SYSTEM (WEBRTC SIGNALING)
================================================================================

Voice rooms are ephemeral and mapped 1:1 to Lobbies.

Signaling Service Responsibilities:
1. Initialize Mediasoup Router for the lobby ID.
2. Manage Client Transports (WebRTC).
3. Handle Producers (Mic streams) and Consumers (Remote listeners).

Signaling Events:
- Request: 'voice:getRouterCapabilities' -> Response: RTP Capabilities.
- Request: 'voice:createTransport' -> Response: Transport ID, ICE Parameters.
- Request: 'voice:produce' -> Notify lobby members: 'voice:newProducer'.
- Event: 'voice:mic_toggle' -> Payload: { userId, muted: boolean }.
- Event: 'voice:vad_update' -> Payload: { userId, speaking: boolean }.

================================================================================
7. PERMISSIONS & ROLES
================================================================================

Lobby Roles:
- HOST: Can start match, kick players, lock room, transfer ownership.
- PLAYER: Can toggle ready, join voice, chat.
- SPECTATOR: Read-only access to lobby state and chat.

Membership Tiers:
- NONE: Standard access.
- PLUS: Blue name, chat glow, exclusive reactions.
- VIP: Golden name, animated badge, priority lobby listing, exclusive frames.

Social Restrictions:
- BLOCKED: Target user is invisible and cannot communicate with blocker.
- PRIVATE LOBBY: Requires 'password' or direct 'invite_id' for entry.

================================================================================
8. NOTIFICATION ARCHITECTURE
================================================================================

Delivery Pipeline:
1. Event Trigger (e.g., Friend Request Sent).
2. Service checks if Peer is online (Redis Presence).
3. If Online: Emit Socket.io event 'notify:received'.
4. Persistent Write: Insert record into `notifications` table for unread state.
5. UI Badge: Target client increments red-dot counter.

================================================================================
9. RANKING & XP ENGINE
================================================================================

XP Values:
- Lobby Join: 5 XP
- Lobby Create: 10 XP
- Daily Chat Activity: 3 XP
- Voice Participation: 7 XP
- Friend Invite: 20 XP

Processing:
- Async workers process `xp_logs`.
- Leaderboards are calculated via Redis Sorted Sets (ZSET) for Weekly/Monthly.
- Chron Job (Redis): Resets weekly rankings every Monday at 00:00 UTC.

================================================================================
10. RATE LIMITING & ABUSE PROTECTION
================================================================================

- Chat Anti-Spam: Max 5 messages / 2 seconds. Penalty: 10s auto-mute.
- Auth Brute-Force: Max 5 login attempts / 15 mins. Penalty: 30m account lock.
- Friend Requests: Max 20 pending requests per user.
- Lobby Flooding: Max 3 lobby creations per minute per user.
- GIF Search: Rate limited to 10 requests / minute.

================================================================================
11. SECURITY ARCHITECTURE
================================================================================

- Encryption: TLS 1.3 for all traffic.
- Password Safety: Argon2id with unique per-user salt.
- JWT Lifecycle: 15m Access Token (Memory), 7d Refresh Token (HttpOnly Cookie).
- Token Rotation: New refresh token issued on every use; old token blacklisted in 
  Redis immediately.
- Input Handling: XSS sanitization for all Persian strings. SQL Injection 
  prevention via Parameterized Queries.
- WebRTC: DTLS-SRTP enforced. TURN over TLS for restricted networks.
